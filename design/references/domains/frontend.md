# 前端工程设计包（Frontend Domain Pack）

> v1 精装版。给设计顾问用——不是教条"必须这样"，是**摆选择、给坑、指好处**，让用户照自己项目情况选。

---

## 一、顶层结构：两大路线，各有坑

### 路线 A：按技术层拆（Horizontal Layers）

```
src/
  components/     ← 纯 UI 组件
  pages/          ← 路由页面（组装 & 编排）
  services/       ← 数据/API 层
  hooks/          ← 跨组件逻辑
  store/          ← 全局状态
  utils/          ← 纯工具
```

- **好处**：一目了然，新人上手快。
- **坑**：业务一多，每一层都膨胀。`components/` 里几十个完全不相关的文件摆在一起，改一个 feature 要跨四五层翻文件。

### 路线 B：按业务 feature 拆（Vertical Slices）

```
src/features/
  strategy/
    components/
    hooks/
    services/
  profile/
    components/
    hooks/
    services/
```

- **好处**：一个 feature 修改只在一个目录下动，边界清。
- **坑**：feature 之间容易重复写同样的东西；共享的东西放哪（`shared/` / `common/`？）需要提前约定。

### 实际多数项目怎么做：**混用**

技术层负责**全局共享的基础设施**：`components/Button`、`services/http`、`hooks/useAuth`、`store/`。业务 feature 各建目录，内部按需分层。一个页面组件同时用全局共享的基础和本 feature 私有部分。

## 二、数据分两桶：Server State vs Client State

**核心区分**：从后端来的数据 vs 只活在 UI 里的信息。混在一起 → 难测、难追踪、难重构。

### Server State（后端来的、需同步的）
- **建议**：统一**数据访问层**收口——所有 API 调用走一个层，页面不直接 `fetch`。
- **配合工具**（react-query / swr / axios 拦截器）：用好缓存键、自动失效、乐观更新、轮询——不是每个页面各写一套。
- **信号**：发现"同一个接口在三个页面各写了一遍"——该抽。

### Client State（UI 开关、表单、弹窗）
- **原则**：能局部就局部，别动不动上全局状态。
- **信号**：store 里有 `isModalOpen`、`formDraft`、`"三个页面的短暂 UI 状态挤在一块"`——该瘦身。

---

## 三、全局状态怎么管

**该放全局的**：用户信息、通知/消息、权限、主题——跨页面、跨模块都需要的。

**不该放全局的**：临时表单草稿、弹窗开关、单个页面内部的计算结果——放局部就行。

**全局 store 在膨胀的信号**：
- store 文件超过 300–400 行
- 里面有"只一个页面在用"的字段
- 每次改 store 不知道会影响谁
- 拆法：按领域切 slice（userStore / notificationStore / permissionStore），各管一块，不互相直改内部状态。

---

## 四、请求层：别散落各处

**统一**：所有 HTTP 调用从**一个地方**发出去，避免每个页面各写一个 `fetch`。

放哪：`services/api` 或 `lib/http`，带拦截、鉴权、错误统一处理。

**怎么抽**：
- 每个 API 域（auth / user / notification）一个模块，里面只有"接口定义 + 参数类型 + 返回值类型"——不写 UI 逻辑。
- 调用方拿到数据或错误，在页面层处理——不把请求和 UI 渲染写混。

---

## 五、页面组件的边界

**页面组件只管两件事**：**组装**（把子组件拼起来）+ **编排**（调 hook/service 拿数据，传给子组件）。业务逻辑抽到 `services/hooks`——不在页面里直接写判断、转换、算法。

**好处**：要换一个页面的整体数据来源？不改组件内部，只改 services 层。

**拆的自然路径**：
1. 先识别"页面文件里哪几块是一眼能看出来的区域"→ 抽成子组件。
2. 看"哪些是纯计算/数据转换/调接口"→ 抽到 services/hooks。
3. 看"还剩下哪些是纯 UI + 数据注入"→ 留在页面本身。
4. 拆完页面只剩"拼装"——职责简单，好改。

---

## 六、依赖方向：单向，别成环

**页面 → services/hooks → 数据访问层 → HTTP**。下层不知道上层存在。

**跨 feature 通信**（feature A 需要 feature B 的东西）：
- **优先**：提取共享接口/模块（两个 feature 都依赖的公共模块）。
- **次选**：全局状态——但只放真正跨模块的数据。
- **避免**：featA 直接 import featB 的内部文件——这会让两个 feature 死死绑在一起；以后要独立修改 featB 时，featA 也跟着坏。

---

## 七、典型反模式

| 反模式 | 为什么是问题 | 怎么改 |
|---|---|---|
| 页面组件写业务逻辑 | 无法复用、难测试、改页面=改逻辑 | 抽到 services/hooks；页面只组装 |
| 请求散落各处 | 鉴权/错误处理重复、缓存不统一 | 统一数据访问层；配合 react-query/swr |
| 全局状态过重 | 任意地方都能改 store、难追踪 | 按领域切 slice；明确"什么才能进全局" |
| `shared/` 变垃圾桶 | 什么都往里扔、边界模糊 | 只放"跨领域真正的共享基础"；目录下也能分模块 |
| 跨 feature 随意 import | 两个模块死死绑住、一改全坏 | 提供公共接口或共享模块；不引用内部实现 |

## 九、领域包视角（设计顾问怎么拿它说事）

对着一个真实前端项目，打开这些镜头依次往下看：
1. 它现在的文件结构——走的路线 A、B、还是混用？每种选择的坑它踩了没？
2. 请求——散落还是一层收口？server state 和 client state 清不清楚？
3. 全局 store——管了哪些东西？有膨胀信号吗？
4. 页面组件——是"组装"还是在里面写逻辑？
5. 跨 feature 引用——有没有 featA 直接 import featB 内部的？
6. 有 `shared/` 吗——里面是干净的还是垃圾桶？

**不强制任何答案。只把现实摆出来、把坑指出来、把选择摊给你。**
