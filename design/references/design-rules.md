## 7 条通用工程原则（怎么想——设计顾问的内部镜头）

逐条带提问脚本，扎在用户真实项目上说。

### 1. 单一职责 / 边界
- 这个模块/文件，能用**一句话**说清它干啥?说不清 → 边界不清 → 该拆。
- 提问："如果我让你一句话告诉新人这个模块是干嘛的，你能说清吗?"

### 2. 对外接口 vs 内部实现
- 能改内部实现而**不惊动用它的人**吗?不能 → 接口暴露了内部细节。
- 提问："外部只通过什么和这个模块说话?能不能把'怎么做的'藏起来?"

### 3. 全局 / 横切关切先统一
- 配置、通知消息、日志、错误处理、鉴权——这些是**全局**的。先定**一个**统一入口，别散落各模块自写一套。
- 提问："你的通知/配置/日志——是各写各的，还是有个统一入口?"

### 4. 别报流水账
- 不相关逻辑别塞进同一个文件;一个文件干了太多事 → 拆积木。
- 提问："这个文件里，有不属于它(一句话职责)的东西吗?"

### 5. 依赖单向、别成环
- 谁依赖谁想清楚;别 A→B→A。分层的别跨层直连。
- 提问："能不能画一条线，所有依赖都只往一个方向?"

### 6. 命名如其实
- 名字说清楚它是什么 / 干什么。

### 7. 先积木、后填充
- 先定有哪几块(模块)、块和块之间怎么拼(接口)——**然后再动手填每块内部**。不是打开文件就写。
- 提问："我们先把积木摆好再动手，你现在有哪几块?"

## 两套 scope 清单

### Project scope

1. **先搞清楚这项目到底是干嘛的**(复用 setup 的 context：干嘛的/谁用/最担心啥)。
2. 过 7 条原则，逐条落在项目上 → 提议：
   - **顶层模块划分**(模块地图:各管什么)
   - **全局关切** 谁统一负责(配置/通知/鉴权/日志)
   - **依赖方向**
   - **公共接口**(模块对外暴露什么)
   - **放哪、分层约定**
3. 提议一份**宪法**:把这个项目不可谈判的规矩写成具体规则。
4. 输出:结构建议 + 宪法草案。**不建目录、不动文件。**

### Module scope

1. **这个功能干嘛**(一句话职责)。
2. 过 7 条：它的**单一职责**是啥?**对外接口**暴露什么?**内部**分哪几块?碰哪些**全局关切**?放进现有哪块还是新起?会不会变**流水账**?命名合适吗?**先想积木再动手?**
3. 特别问：**这个模块不负责什么**(负向边界，逼清界线)。
4. 对照宪法：违不违宪?
5. 输出:该模块的结构建议(放哪、接口长啥样、碰哪些全局、风险)。**不写代码。**

## 宪法（Constitution）—— 跨 A/B 的桥

- 位置：`.project-health/constitution.yml`，YAML，用户确认后写。
- Schema：
```yaml
schema_version: 1
rules:
  - id: fe-page-no-business-logic
    scope: module
    applies_to: "src/pages/**"
    rule: "页面组件不写业务逻辑，抽到 services / hooks"
    level: hard              # hard(🔴) | advisory(⚠️)
    reason: "页面一旦掺业务逻辑，复用和测试都受阻"
  - id: max-file-lines
    scope: project
    rule: "单个源文件不超过 400 行"
    level: advisory
    reason: "过大文件改动风险高"
```
- **`level`**：`hard` → 未来 A 面 audit 报 🔴；`advisory` → ⚠️。
- **只收**：经用户确认、长期稳定、**具体可判断**的规则。**不收抽象口号**(那 7 条是内部镜头，不写进宪法)。
- **和 `config.yml` 的关系**：宪法是"规矩"，config 是"设定"；不重叠。config 里的 `project_rules` 字段收敛到宪法（单一真源）。

## 前端领域包（v1 薄样板）

位置：`design/references/domains/frontend.md`。

只写：
- **默认分层约定**：`components/`(纯展示) · `pages/`(路由) · `services/`(数据/接口) · `hooks/` · `store/`(全局状态) · `utils/`——给默认、不教条。
- **全局关切**：全局状态/配置/请求层放哪。
- **依赖方向**：页面→services→接口；别反向。页面不直接调 HTTP 客户端。
- **典型反模式**：
  - 页面组件写业务逻辑
  - 全局状态塞进页面
  - 请求散落各处
  - `shared/` 变成垃圾桶
  - 跨 feature 依赖无限制
- **权威引用**（只指路）：框架官方结构指南、`create-*` 脚手架约定。

对**无深度包**的领域 → **诚实回落**到 7 条通用原则，并明说"我没有该领域的深度包"。
