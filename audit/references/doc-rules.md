# doc-rules.md — 文档侧检查规则（C2 断链 / C3 超大文档）

> audit v1。确定性检查。**C2 的第一原则：宁漏勿错**——误报一个断链，用户就不再信任整个工具。

---

## 扫描哪些文档

- 仓库根：`CLAUDE.md`、`AGENTS.md`、`README.md`
- `docs/**/*.md`
- （其它 `*.md` 本版不主动扫，避免噪声。）

---

## C2 · 断链（路径 + 命令）

对每篇文档，逐行提取"引用"，判断目标是否存在。

**预处理（重要）**：先剔除 **fenced code block**（一对 ` ``` ` 之间的内容）再提取——代码块里的 `[..](target)`、`path#Lline` 多是示例/模板，不是真链接。跳过它们能消掉最常见的一类误报。

### 一、路径引用（保守提取）

**提取以下三类，其余一律不提取**：
1. **Markdown 链接** `[文字](目标)` 中，`目标`为本地相对路径。
   - 排除：`http://`、`https://`、`mailto:`、以 `#` 开头的纯锚点。
   - 若目标含 `#anchor`，只取 `#` 前的路径部分。
2. **反引号 token** 且**明确像仓库路径**——满足任一：
   - 以 `./` 或 `../` 开头；或
   - **第一段是仓库中真实存在的顶层目录**（先 `ls` 仓库根拿到顶层目录集合，token 的第一段命中才算路径）。
3. 仅此两类。**不提取**：无路径的裸文件名（如正文里的 `service.py`）、明显是示例/占位的路径（如 `path/to/xxx`、`your-project/`）。

**存在性判定**：目标依次尝试
- 相对**仓库根**解析 → `test -e`
- 相对**该文档所在目录**解析 → `test -e`
两者都不存在 → 🔴 断链。任一存在 → 通过。
（目标以 `/` 结尾视为目录，判目录存在。）

**输出每条**：`<文档>:<行号> → <引用目标>` + "指向不存在的路径"。

### 二、命令引用（仅 npm 系，且项目有 package.json 时）

- 若仓库根**无 `package.json`** → 跳过命令检查（不报错）。
- 提取形如 `npm run <script>` / `pnpm run <script>` / `yarn <script>` / `pnpm <script>` 的命令（代码块或反引号内）。
- `<script>` 不在 `package.json` 的 `scripts` 键中 → 🔴 "文档引用了不存在的 npm 脚本"。
- mvn/gradle/make 等其它命令体系：**本版不检查**（留给完整 Stage 1）。

**输出每条**：`<文档>:<行号> → <命令>` + "引用了不存在的脚本"。

### C2 防误报红线
- 顶层目录集合必须来自**实际 `ls`**，不是猜。
- 拿不准的一律**不报**。宁可漏一个真断链，也不冤枉一个正常词。

## C3 · 超大文档

- **对象**：C2 扫描的同一批文档（根三件 + `docs/**/*.md`）。
- **度量**：总行数（含空行即可，文档空行本身也占篇幅）。
- **判级**（config `thresholds.doc_warn`，默认 500）：
  - 行数 ≥ 500 → ⚠️。
- **输出每条**：`<文档>:<行数>` + "文档过长，建议拆分/压缩；已被迭代淘汰的旧记录可归档为一句话摘要"。
