# fix-rules.md — 各类型修复细则（F1 断链 / F2 压缩归档 / F3 拆分）

> Stage 2。fix **会写文件**，每条规则都以安全为先。分级见 SKILL.md。

---

## 通用：改前 / 改后

**改前**：
- `git status` 确认工作区干净。不干净就先让用户提交/搁置已有改动，别混进 fix 的 commit。
- 项目**有测试**：先跑一遍记基线（记住哪些通过）。命令从 `package.json`/`pom.xml`/`Makefile` 等识别。

**改后**：
- **重跑 audit 的对应检查**，确认该项已消失。
- 有测试：再跑一遍，确认没弄坏（对比基线）。
- **一项一 commit**，信息格式：`Fix(audit-<类型>): <一句话> — <文件>`。

---

## F1 · 文档断链修复（🟢 机械，直接做）

对 audit 报出的每条断链：
1. **先找正主**：用搜索（按文件名 / 就近目录）找目标是否只是**移动/改名**了。
   - 找到 → 把文档里的路径**改成新路径**。
2. **确实没了**：目标已删除、无对应物 →
   - 若这句话没了引用就不成立 → **删掉该引用**（或整行）。
   - 若只是举例 → 保留文字但去掉链接。
   - 不确定该删还是该改 → **停下问用户**，别猜。
3. **命令断链**：`npm run <script>` 但 `package.json` 没有 → 改成真有的脚本名；拿不准就问。

> 红线：宁可**问一句**，也不乱改文档语义。

## F2 · 文档压缩归档（🟢 机械，直接做）

两种情况：

**(a) 超长文档拆分**（audit C3）：
- 按主题把一篇长文拆成多篇 → 原文件留一个**薄索引**（一句话 + 指向各子文档的链接）。
- 拆分**不丢内容**，只是重新组织。

**(b) 淘汰版本记录压缩**（呼应 blueprint §十）：
- 找出**已被完全淘汰版本**的逐条 bug/修改记录。
- 压成**一句话摘要**："曾有 vX，大致功能 Y"。
- 原详细内容**移入 `archive/`**（新建或沿用），**不是删除**——可回溯。
- 活文档只保留当前有效内容。

> 红线：**归档 = 搬走存起来，不是丢掉**。任何"删"都要能在 archive 里找回。

## F3 · 大文件拆分（🟡 重构，先方案后确认）

**绝不直接拆。** 必须走四步：

**① 出方案**（给用户看，等确认）：
```
待拆：<file>（<行数> 行）
现状：这个文件现在管了哪几件事（列 3-6 条职责）
建议拆成：
  - <新文件A>：职责……
  - <新文件B>：职责……
影响面：哪些文件 import/引用了它，拆完要跟着改哪些
风险：<如私有方法交织、测试覆盖不足等>
验证：拆完跑哪些测试 + 重跑 audit
```
**② 等用户确认**（同意 / 调整 / 放弃）。放弃就转 suppressions（见下）。

**③ 执行**（确认后）：
- 小步搬：一次搬一块职责，随手更新引用。
- **不改行为**，只挪位置（纯重构）。
- 保护面（接口/DB/路由）不动。

**④ 验证**：跑测试（对比基线，必须全绿）+ 重跑 audit（该 🔴 应消失或降级）。

> 若项目**没有测试**：明确告诉用户"拆分缺少测试兜底，风险偏高"，让 ta 决定是否继续。

## 写 suppressions（"这项我认了，先别修"）

用户说某项先放着 → **不改代码**，在 `.project-health/config.yml` 加：
```yaml
suppressions:
  - id: "<check>:<repo相对路径>"     # 如 large-file:src/Foo.java
    reason: "<为什么暂不修>"
    expires: "YYYY-MM-DD"           # 可选；到期后 audit 会再提醒
```
- 无 config 文件就新建一个最小的。
- 这是 audit 只读 suppressions 的**写入端**；写完下次 audit 即生效。

---

## commit 规范

- **一项一 commit**，别把多项塞一个提交。
- 信息：`Fix(audit-broken-ref|oversized-doc|large-file|archive): <一句话> — <文件>`。
- F3 拆分可能涉及多文件，但仍是**一个逻辑项 = 一个 commit**（或一组紧邻的小 commit，说明清楚）。
