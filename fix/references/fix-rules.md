# fix-rules.md — 各类型修复细则（F1 断链 / F2 压缩归档 / F3 拆分）

> Stage 2。fix **会写文件**，每条规则都以安全为先。分级见 SKILL.md。

---

## 通用：改前 / 改后

**改前（基线）**：
- `git status` 确认工作区干净。不干净就先让用户提交/搁置已有改动，别混进 fix 的 commit。
- **先读 `execution.trust`** 和 `approved_verify`（见 §验证机制）。
- 文档类修复（🟢）：不用 verify。

**改前（代码类）——记录 touched files 清单 + 跑基线**：
- 记录本次将要修改/新建的文件路径（touched files），包括**新建的未跟踪文件**。
- 若 trust 允许：跑 verify 命令 + 测试作为基线。基线**不全绿** → 记录失败集合（不改前就不通过），不一刀切禁止（很多存量项目本就带失败）；**基线全绿** → 改后必须不新增失败。

**改后**：
- **重跑 audit 的对应检查**，确认该项已消失。
- **若 trust 允许、且改代码了**：再跑一遍 verify + 测试。
  - 都通过 → 提交。
  - baseline 全绿、改后失败 → fix 改坏了 → **精确回滚**（只还原 touched files）。
  - baseline 本有失败 → 比失败集合：**新增失败 = 改坏了 → 回滚**；旧失败照旧 = OK。
  - 失败集合无法可靠比较 → **说"无法自动确认"，不假装验证过。**
- **一项一 commit**，信息格式：`Fix(audit-<类型>): <一句话> — <文件>`。

## 验证机制（execution.trust + verify + rollback）

### 优先级（高→低）
`approved_verify` 白名单 > `execution.trust`（disabled / prompt / trusted）> 自动探测命令。

### 三档 trust

**`disabled`**：**不执行任何项目 build/test/verify/typecheck 命令**。只读检查（re-audit、读文件、diff）仍然允许。改后明确告诉用户："安全闸门关着，未执行自动验证——建议你手动确认项目还能正常 build/跑通。"

**`prompt`**（默认）：首次想跑某个 verify/test 命令前，**先展示具体命令 + 说明会执行项目代码 → 等用户确认 → 确认后才跑**。确认后可记入 `approved_verify`（下次不再问）。**Agent 不自行往白名单加东西。** 自动探测出的命令**不算"已批准"**——必须单独确认。

**`trusted`**：可直接执行 `config.verify` 和已在 `approved_verify` 白名单里的命令。自动探测出的新命令**仍需确认或与白名单对照**。

### verify 命令来源
1. `config.verify`（用户显式指定，最准）。例：`verify: "npm run build"`
2. 自动探测（仅在 trust 允许时）：`npm run build` / mvnw compile / gradle / Python compileall
3. 都没有 → 诚实说"无法自动确认没改坏"。

### 精确回滚
- **改前记录 touched files**(含新建文件)。
- 改坏了 → **只还原/删除 touched files**:
  - 改过的已有文件 → `git checkout -- <file>` 或 `git restore <file>`
  - 本次新建的文件 → 直接删除
  - **不碰**：`git restore .` / `git checkout -- .` / `git clean -fd` / `git reset --hard`
  - **不碰**：原有未跟踪文件（不属于本次 fix 的）
- 若无法确定精确回滚范围 → **停下、报告**，不执行宽泛清理。

### 基线本有失败的处理
- 改前记录失败集合(文件名/测试名/错误摘要——能可靠比较才行)。
- 改后比对：**只新增的失败**算 fix 引入的 → 回滚；**旧失败照旧** → 允许。
- 若失败集合**无法可靠比较**(非结构化输出、随机数据、不可靠的测试)→**说"无法自动判断是否新增失败"，不装。**

### config 字段
```yaml
verify: "npm run build"
execution:
  trust: prompt          # disabled | prompt | trusted
  approved_verify: []    # 用户批准过的命令白名单
```
- `doc_links` 也在 config 中（watch 用），fix 不直接读，但保持契约一致。

## F1 · 文档断链修复（🟢 机械，直接做）

对 audit 报出的每条断链：
1. **先找正主**：用搜索（按文件名 / 就近目录）找目标是否只是**移动/改名**了。
   - 找到 → 把文档里的路径**改成新路径**。
2. **确实没了**：目标已删除、无对应物 →
   - 若这句话没了引用就不成立 → **删掉该引用**（或整行）。
   - 若只是举例 → 保留文字但去掉链接。
   - 不确定该删还是该改 → **停下问用户**，别猜。
3. **命令断链**：`npm run <script>` 但 `package.json` 没有 → 改成真有的脚本名；拿不准就问。

> 红线：宁可**问一句**，也不乱改文档语义。

## F2 · 文档压缩归档（分两种，风险不同）

### F2a · 纯归档搬移（🟢 直接做）

把**已被完全淘汰版本**的逐条记录搬走存档、**不改正文语义**：
- 找出淘汰版本的逐条 bug/修改记录。
- 原详细内容**移入 `archive/`**（新建或沿用），**不是删除**——可回溯。
- 正文对应处压成**一句话**："曾有 vX，大致功能 Y"。

> 红线：**归档 = 搬走存起来，不是丢掉**。任何"删"都要能在 archive 里找回。

### F2b · 重排 / 压缩 / 拆分文档结构（🟡 先方案后确认）

凡是**改变一篇（重要）文档的结构或内容组织**——如把长文按主题拆成多篇、重写章节、大幅压缩正文——**不直接做**：
1. 先给**简短方案**：拆成哪几篇 / 每篇放什么 / 原文件留什么薄索引 / 影响哪些引用。
2. **等用户确认**（同意 / 调整 / 放弃）。
3. 确认后执行：拆分**不丢内容**，只重组；原文件留一个薄索引（一句话 + 指向各子文档的链接）。

> 为什么收紧：重要文档**怎么组织，用户往往有偏好**；别擅自大改结构。

## F3 · 大文件拆分（🟡 重构，先方案后确认）

**绝不直接拆。** 必须走四步：

**① 出方案**（给用户看，等确认）：
```
待拆：<file>（<行数> 行）
现状：这个文件现在管了哪几件事（列 3-6 条职责）
建议拆成：
  - <新文件A>：职责……
  - <新文件B>：职责……
影响面：哪些文件 import/引用了它，拆完要跟着改哪些
风险：<如私有方法交织、无测试兜底等>
验证：跑 verify 命令（构建/编译）+ 测试 + 重跑 audit
```
**② 等用户确认**（同意 / 调整 / 放弃）。放弃就转 suppressions（见下）。

**③ 执行**（确认后）：
- 小步搬：一次搬一块职责，随手更新引用。
- **不改行为**，只挪位置（纯重构）。
- 保护面（接口/DB/路由）不动。

**④ 验证**：跑 **verify 命令（构建/编译）** + 测试（对比基线，必须全绿）+ 重跑 audit（该 🔴 应消失或降级）。**基线过、改后不过 → 立即撤销本次拆分**（见"验证机制"）。

> 若项目**既没测试、也没有可用的构建命令**：明确告诉用户"缺少任何自动兜底，风险偏高"，让 ta 决定是否继续（并建议先配 `config.verify` 或手动跑一遍）。

## 写 suppressions（"这项我认了，先别修"）

用户说某项先放着 → **不改代码**，在 `.project-health/config.yml` 加 suppression。

**ID 直接从 audit 报告/latest-run.yml 里拿 finding 的 `id` 原值，不要自己拼**（scanner 生成的稳定格式是 `C1|large-file|src/Foo.java`、`C2|broken-ref|README.md|docs/missing.md` 等——scanner 用竖线分隔，和 fix 文档旧写法不同。自己拼出来的 id 匹配不上 → suppression 不生效 → 下次继续报警）。
```yaml
# 示例（每条的 id 直接拷自 finding.id，不自己造）：
suppressions:
  - id: "C1|large-file|src/Foo.java"
    reason: "已评估，当前阶段暂不拆分"
    expires: "2026-09-01"          # 可选
```
- 无 config 文件就新建一个最小的。
- 写完下次 audit 自动生效（scanner 读 suppressions 按 `id` 精确匹配）。

## commit 规范

- **一项一 commit**，别把多项塞一个提交。
- 信息：`Fix(audit-broken-ref|oversized-doc|large-file|archive): <一句话> — <文件>`。
- F3 拆分可能涉及多文件，但仍是**一个逻辑项 = 一个 commit**（或一组紧邻的小 commit，说明清楚）。
