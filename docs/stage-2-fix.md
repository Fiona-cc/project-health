# Stage 2 · fix — 实现 Spec

> 状态：待审
> 日期：2026-07-10
> 上位：`blueprint.md`（§四 子 skill 职责、§十 文档演进）
> 定位：拿着 audit 的报告，**在你点名某项之后**，把那一项**修掉**；一项一个 commit，修完重跑 audit 确认。

---

## 一、目标与范围

**一句话**：audit 只"找问题"，fix 负责"**动手解决**"——但**会写代码/文档**，所以**安全第一**。

**v1 处理这几类修复**：

| ID | 修什么 | 来自 audit 的 | 风险 |
|----|--------|--------------|------|
| F1 | 文档断链修复（改正/删除失效路径、改正命令） | C2 | 🟢 低 |
| F2a | 文档归档搬移（淘汰版本记录搬进 archive，**不改正文语义**） | §十 | 🟢 低 |
| F2b | 文档结构重排/压缩/拆分（改变重要文档的组织） | C3、§十 | 🟡 中（先方案后确认） |
| F3 | 大文件拆分（超大源文件按职责拆） | C1、C4 | 🟡 高 |
| — | 写 suppressions（"这项我认了，别修"→ 记进 config，不改代码） | — | 🟢 低 |

---

## 二、核心安全原则（因为它会动手，这是重点）

1. **人说才修**：**绝不自动批量修**。只修用户**明确点名**的项（"修第 3 项""把文档断链都修了"）。
2. **一项一 commit**：每项独立提交，随时能**单独回滚**。
3. **改前留底**：先确认工作区干净（或先提交/搁置你已有的改动），别把 fix 和你别的改动**混在一个 commit**。
4. **改完复查**：修完**重跑 audit 对应项**确认它消失；项目**有测试的**，改前跑一遍（基线）、改后再跑一遍（确认没弄坏）。
5. **不扩大战场**：只动与该项**直接相关**的文件，不顺手改别的。
6. **拿不准就停**：吃不准怎么修、或风险大，**先给方案让用户拍板**，不擅自动手。
7. **不碰受保护面**：HTTP 接口 / DB schema / 前端路由等，除非用户明确要求并确认（沿用 GiftBook `AGENTS.md` 的保护约定）。

---

## 三、分级修复（按风险，红黄绿）

| 级别 | 类型 | 怎么修 | 谁先点头 |
|------|------|--------|---------|
| 🟢 机械 | F1 文档断链 | 找到正确路径就改；找不到对应物就标注/删除；命令改成 `package.json` 里真有的 | 直接做，做完给你看 diff |
| 🟢 机械 | F2a 归档搬移 | 淘汰版本记录**移入 archive**、正文压成一句话，**不改语义** | 直接做，做完给你看 diff |
| 🟡 重构 | F2b 结构重排/压缩/拆分 | 改变重要文档的组织（拆多篇/重写/大幅压缩）→ **先给简短方案** → 确认后执行 | **先方案，你点头再动** |
| 🟡 重构 | F3 大文件拆分 | **必须先出拆分方案**（拆成哪几块、每块什么职责、动哪些引用）→ **你确认** → 改 → 跑测试 + 重跑 audit | **先方案，你点头再动** |

> 🟢 可以直接做；🟡 **必须先给方案、你确认后再动手**。这条是 fix 安全的核心闸门。

---

## 四、逐项流程

```
你说"修第 X 项"（或"把文档断链都修了"）
   │
   ▼
① 定位：从最近一次 audit 报告拿到该项的 file:line 和类型
② 判级：🟢 机械 → 直接修 ；🟡 重构 → 先出方案、等你确认
③ 改前：确认工作区干净；有测试则先跑一遍（记基线）
④ 修：只动相关文件
⑤ 验：重跑 audit 该项（确认消失）＋ 有测试再跑一遍（确认没弄坏）
⑥ 提交：一项一个 commit，信息写清"修了啥、来自 audit 第几项"
```

---

## 五、suppressions（"这项我认了，先别修"）

- 用户说"第 X 项先放着、不用修" → **不改代码**，改为在 `.project-health/config.yml` 的 `suppressions` 里加一条：`id` + `reason` + 可选 `expires`。
- 下次 audit 就不再报它（挪进"看着吓人其实没事"）。
- 这是 audit 里**只读的 suppressions 的写入端**（Stage 1 只读、Stage 2 负责写）。

---

## 六、压缩归档（F2 细化，呼应 §十）

- **目标**：文档一版版迭代后，旧版本的逐条 bug/修改记录已经没意义了。
- **做法**：把**已被完全淘汰版本**的详细记录 → 压成一句话（"曾有 vX，大致功能 Y"）→ **移到 `archive/`**；活文档只留当前有效内容。
- **安全**：原文**不是删掉，是移进 archive 存档**（可回溯），不是丢弃。

## 八、本版明确不做

- 不自动批量修、不"一键全修"。
- 不碰 audit 报告**之外**的问题（不顺手改）。
- F3 复杂重构**不追求全自动**——只做"方案 → 确认 → 执行 → 验证"，真正的拆分判断仍要人参与。
- 不动受保护面（接口/DB/路由），除非明确要求且确认。

---

## 九、验收标准（在**一次性测试项目**上验，不碰任何真实项目）

> fix 会**写文件**，所以验收**绝不在 GiftBook 等真实项目上做**。而是造一个**一次性测试项目**（临时文件夹，故意埋几个问题），验完即删——既证明 fix 能用，又**不动你任何真项目**。

- **造 fixture**：一个临时目录，含 ①一篇文档里写个失效路径、②一篇 600 行的超长文档、③一个 900 行的假源文件。
- **F1**：修那个失效路径 → 重跑 audit 该断链消失 → 一个干净 commit。
- **F2**：把那篇超长文档压缩/归档 → audit 该项消失、原文进 `archive/`。
- **F3**：对那个 900 行假文件**出一份拆分方案**（验证"先方案后确认"流程，不必真拆）。
- 全程**一项一 commit、可回滚**；验完**删除 fixture**。
