## 一、目标与范围

**一句话**：一个**域无关**的、**可缩放**（项目/模块两个 scope）的设计顾问。核心是**通用工程原则**（跨领域）；前端是 v1 唯一带"深度领域包"的领域，其它领域**诚实回落到通用原则**。

**v1 做这些**：
1. **通用工程思考框架**（7 条，域无关，是顾问"怎么想"的内部镜头）——§四。
2. **两个 scope**：project / module，**先识别 scope，再用各自不同的清单**——§六。
3. **工程宪法**：用户确认后写/更新 `.project-health/constitution.yml`（机器可读，A/B 共享的桥）——§五。
4. **一个薄前端深度包** + 一个**领域包模板**——§七。

**输出纪律**：一次性设计建议**默认只在对话里展示**，不为每个任务堆一篇 md；**唯一落盘的持久物是宪法**，且**用户确认后才写**。

## 三、核心原则

1. **只建议、不动手**：不创建业务代码/目录；唯一写的文件是 `constitution.yml`（用户确认后）。
2. **建议默认在对话里**：别每个任务堆一篇 markdown（避免"Markdown 疯魔"）。
3. **域无关内核 + 诚实回落**：通用 7 条谁都能用；有深度包的领域（v1=前端）给额外增强；**没深度包的领域,明说"我没有 X 的深度骨架,给你通用原则",绝不编造深度支持**。
4. **深处通用、浅处引用**：领域包薄，指向权威（不抄百科）。
5. **顾问不替判断，且不讲空话**：它摆出"该想的点"逼你想清楚；**建议必须扎在你项目的真实结构上给具体的**（"你 X 现在同时管了 A/B/C,建议拆成…"），不是背通用清单。
6. **猜后再问**：复用 setup 的检测（领域/技术栈/context），别让用户从零描述。

## 四、通用工程思考框架（7 条，域无关，**内部镜头**）

> 这 7 条是顾问"怎么想"的框架，**不直接抄进宪法**（宪法只收由它们推导出的**具体规则**，见 §五、§六原则）。

1. **单一职责 / 边界**：能用一句话说清它干嘛?说不清 → 拆。
2. **对外接口 vs 内部实现**：能改内部而不惊动用它的人吗?不能 → 接口漏了细节。
3. **全局 / 横切关切先统一**：配置、通知、日志、错误处理、鉴权——先定统一入口，别各写各的。
4. **别报流水账**：不相关逻辑别塞一个文件;一个文件干太多 → 拆。
5. **依赖单向、别成环**：别 A→B→A;分层别跨层直连。
6. **命名如其实**。
7. **先积木、后填充**：先定有哪些块、块怎么拼（接口），再填每块内部。

## 五、工程宪法（Constitution）—— A/B 的机器可读桥

- **位置 / 格式**：`.project-health/constitution.yml`（**YAML，非 md**——因为 A 面要能"照它查"）。
- **每条规则字段**：
  ```yaml
  rules:
    - id: fe-no-logic-in-page
      scope: module            # project | module
      rule: "页面组件不写业务逻辑，抽到 services / hooks"
      level: hard              # hard（违反=🔴） | advisory（违反=⚠️）
      reason: "页面组件一旦掺业务逻辑，复用和测试都受阻"
    - id: file-max-lines
      scope: project
      rule: "单个源文件 ≤ 400 行"
      level: advisory
      reason: "过大文件改动风险高、难维护"
  ```
- **`level` 直接接 A 面严重度**：`hard` → audit 报 🔴；`advisory` → ⚠️。（桥就通在这。）
- **只收什么**：**经用户确认、长期稳定、具体可判断**的规则。**不收**抽象口号（"保持单一职责""为变化留空间"）、也不收一次性临时建议。
- **谁写/谁读**：本 skill 确认后写；**A 面 audit 将来读它来查**（本版不建这步，只保证文件立得起来）。
- **与 config 的关系**：config 里原 `project_rules` 字段**收敛为**指向 `constitution.yml`（单一真源，不两处存规矩）。

## 六、两个 scope（一个 skill，**先识别 scope，问不同清单**）

**开场先判 scope**：用户说的是"整个项目/应用"还是"某个功能/模块"?判不准就问一句。

**project scope**（"帮我理这个项目的工程框架"）问/给：
- 模块图（有哪些顶层模块、各管什么）
- 应用级能力（通知/配置/鉴权/日志等横切,谁统一负责）
- 业务边界（模块之间界线）
- 依赖方向（谁依赖谁,别成环）
- 公共接口（模块对外暴露什么）

**module scope**（"帮我设计这个通知功能"）问/给：
- 这个模块的**职责**（一句话）
- **不负责什么**（负向定义,逼清边界）
- **对外接口**（暴露什么）
- **内部结构**（内部分哪几块）
- **依赖**（依赖谁、被谁依赖）
- **状态 / 配置 / 错误 / 测试** 怎么安排
- 放进现有哪块 or 新起一块;**是否违宪**

**流程（两 scope 一致）**：识别 scope → 用对应清单 + 扎在真实结构给具体建议（对话里展示）→ **用户确认/调整** → 若产出稳定具体规则,**确认后写进 `constitution.yml`**。

## 七、前端深度包（薄）+ 领域包模板

- 位置：`design/references/domains/frontend.md`（薄）、`design/references/domains/_template.md`（模板,供 DL 等照抄）。
- **前端包只写**（不写百科、不强制唯一目录）：
  - 默认模块分层（如 `components/ pages(或 routes)/ services/ hooks/ store/ utils/`——**给默认,不教条**）
  - 全局关切（全局状态/配置/请求层放哪）
  - 依赖方向（页面→services→接口;别反向）
  - 典型反模式（页面组件塞业务逻辑、全局状态塞进页面、请求散落各处…）
  - 少量**权威引用**（框架官方结构指南 / create-* 约定——只指路）
- 无深度包的领域 → §三·3 诚实回落。

## 十、验收标准（在 SmartMarketing_demo 真机验，低风险）

B 面**只出建议 + 顶多写一份 `constitution.yml`**（不碰业务代码），真机验很安全：
- **project scope**："帮我给 SmartMarketing_demo 理工程框架" → 产出:顶层模块图/全局关切/依赖方向/公共接口建议（**扎在它真实结构上,不讲空话**）+ 提议若干**具体**宪法规则;**确认后**才写 `constitution.yml`;**不动业务文件**。
- **module scope**："帮我设计一个'通知消息'模块" → 产出:职责 / 不负责什么 / 对外接口 / 内部结构 / 依赖 / 状态·配置·错误·测试 / 放哪 / 是否违宪;**不写代码**。
- **质量红线**：建议必须**具体、扎在真项目**;空泛清单 = 不合格。
- 宪法文件立得起来、字段齐（id/scope/rule/level/reason）、只含具体规则。
