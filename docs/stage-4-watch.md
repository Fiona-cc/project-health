# Stage 4 · watch — 实现 Spec

> 状态：待审
> 日期：2026-07-10
> 上位：`blueprint.md`（§四 职责、§八 baseline、§十 文档演进）
> 定位：长期养护——**立基线，之后只报"增量"**（新增 / 已解决 / 遗留），别每次都把全量问题再吓你一遍。

---

## 一、目标与范围

**一句话**：watch = 带记忆的 audit。第一次**立基线**，以后每次只告诉你**"和上次比，多了什么、少了什么、还剩什么"**——让它像个长期管家，而不是一次次重新吓人的审计器。

**watch v1 做这些**：
1. **基线（baseline）**：首次运行先出结果、**经用户确认后才存成基线**；之后以它为对比参照。
2. **增量对比**：跑一次新 audit → 和基线逐条比 → 标 **新增 / 已解决 / 遗留**。
3. **文档漂移检测**：最近改过的代码文件，其相关文档**可能**没跟上（读 git 时间，只推断、不定论）。
4. **不碰你的代码/项目文档**：只写自己在 `.project-health/` 下的产物（watch 报告；`baseline.md` 需用户确认才写）。

---

## 二、本版明确不做（推迟，但留插座）

| 推迟项 | 去哪 | 为什么 |
|---|---|---|
| **主动文档守护 hook**（"每次操作后主动问要不要维护文档"） | 后续（用 Claude Code hook 实现） | skill 只在**被调用时**才跑，做不到"每次编辑后自动弹"；那需要 `settings.json` 的 PostToolUse/Stop **hook**，是**另一种机制**，不是 skill |
| 定时/后台自动跑 | 后续 | 同理，需外部调度，不属 skill |

> 把"文档守护"拆两半：**检测**（最近代码变了、文档没跟）是 skill 能做的 → watch v1 做；**主动提醒**（操作后自动弹）是 hook → 留插座（config 的 `doc_maintenance.prompt_after_ops` 已预留）。

---

## 三、核心原则

1. **只报增量、别再吓人**：重点是**新增**；已解决给个肯定；遗留只给个数，不重复罗列细节。
2. **不碰你的代码/项目文档**：watch 只读你的项目；只写自己在 `.project-health/` 下的产物（watch 报告每次；`baseline.md` **仅用户确认才写**）。改东西是 fix 的活。
3. **尊重 suppressions**：被 suppress 的项不算"新增"。
4. **基线可更新**：修完一批问题后，可以把当前状态设为新基线（"接受现状"）。

---

## 四、baseline & 增量机制

**基线从哪来**（按顺序）：
1. `.project-health/baseline.md`（显式基线，最稳）。
2. 没有 → 用 `.project-health/reports/` 里**最近一次 audit 报告**当参照。
3. 都没有 → 这是**第一次**：跑一次 audit、**展示结果**，**询问用户是否设为基线**；**确认才写** `baseline.md`，不确认就只展示本次、不自动立基线。

**怎么比**（按发现的稳定 id）：
- 每条发现 id = `<check>:<路径>`（与 suppressions 同一套 id）。
- 跑一次**新 audit** → 得到当前发现集合。
- 与基线集合比：
  - **新增**：当前有、基线没有 → ⚠️ 重点报。
  - **已解决**：基线有、当前没有 → ✅ 报喜。
  - **遗留**：两边都有 → 只给数量 + 一句"仍在，详见上次报告"。

---

## 五、文档漂移检测（"文档守护"的 skill 那一半）

- 读 git：最近 N 天（默认 14）改动过的**代码文件**。
- 找它**相关的文档**（同目录/同模块的 `*.md`、或根 `CLAUDE.md`/`README.md`），看文档的最后改动时间是否明显早于代码。
- 明显滞后 → 提示"这些代码近期改了，对应文档**可能**没跟上，**建议看一眼**"——**不写成"文档未更新"这种定论**（也许文档不用改）。
- **只提示、不改**（改文档是 fix 的活）。

> 主动"操作后弹窗提醒"那种，是 hook，本版不做（见 §二）。

---

## 六、报告格式

内联 + 落盘 `.project-health/reports/watch-YYYY-MM-DD.md`。

```markdown
# 项目健康监控 · <YYYY-MM-DD>（对比基线 <基线日期>）

## 🆕 新增问题（<n>）  ← 重点
- `<path>:<line>` [<类型>] — <一句> — 建议动作

## ✅ 已解决（<m>）
- `<path>` [<类型>]

## 🟰 遗留（<k>，未重复展开）
- 仍有 <k> 项在册，详见基线/上次报告。

## 📄 文档可能未同步（<j>）
- `<doc>` — 相关代码 <code> 近 <days> 天改过，**文档可能未同步，建议看一眼**

## 说明
- 基线：<baseline 来源与日期>
- 忽略/跳过：<同 audit>
```

**没有新增时**：明确写"自基线以来无新增问题"，不注水。

## 七、skill 结构

```
watch/
  SKILL.md            ← 薄入口：取基线 → 跑 audit → 比对 → 报增量 + 文档漂移
  references/
    watch-rules.md    ← 基线来源/优先级 + id 比对规则 + 文档漂移判定 + 报告模板
```
（复用 audit 的检查引擎与 id 约定；复用 config 的 suppressions。）

## 八、验收标准

**一次性 fixture（先验增量逻辑，不碰真项目）**：
- 造 fixture → 跑 watch 第一次 → 立基线。
- 加一个 900 行大文件 + 改一个已有代码文件的时间戳。
- 再跑 watch → 应报：**🆕 新增** 那个大文件；**📄 文档滞后**（若有关联文档）；遗留只给数。
- 删 fixture。

**整条链路真机验证（在 SmartMarketing_demo 上，串起来验全套）**：
```
配置 project-health(setup) → 检查项目健康(audit,立基线)
→ 随便改一两个文件（制造新增）→ 看最近新增(watch，应只报增量)
→（可选）修第X项(fix)
```
一次把 setup 真机触发、watch、整条生命周期全验实。
